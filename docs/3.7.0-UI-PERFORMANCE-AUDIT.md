# OmenCore 3.7.0 UI Performance Audit

## Release Scope Clarification

- This audit started as 3.7.0 architecture planning work.
- Selected low-risk remediations from this audit were intentionally pulled forward into v3.6.2 stabilization.
- The v3.6.2 pull-in focused on low-risk runtime decoupling, hidden-surface dormancy, and render-state dedupe.
- Deeper projection-coordinator and broader architecture work remains tracked for post-v3.6.2 releases.

## Critical Field Evidence

### New report (high severity)
- "Reinstalled after letting the project develop after 6 months. Still the UI runs at 0.3fps and it's nearly impossible to use GUI..."

### Severity classification
- This is a top-tier architecture failure, not a cosmetic defect.
- This confirms the rendering/runtime overhead issue is not an isolated edge case.
- This indicates the runtime can saturate the UI/render pipeline badly enough to collapse interaction quality.

## Investigation Scope

This audit focused on:
- Dispatcher invoke density
- UI-thread queue pressure
- telemetry-to-UI projection cadence
- PropertyChanged churn
- ObservableCollection churn
- timer and polling overlap
- MainViewModel/DashboardViewModel/GeneralViewModel update density
- overlay and tray refresh cadence

## Findings

### Dispatcher and marshaling pressure
- The app has many dispatcher marshal sites across app, services, view models, and controls.
- Main telemetry projection in [src/OmenCoreApp/ViewModels/MainViewModel.cs](../src/OmenCoreApp/ViewModels/MainViewModel.cs) is already coalesced.
- Dashboard projection path in [src/OmenCoreApp/ViewModels/DashboardViewModel.cs](../src/OmenCoreApp/ViewModels/DashboardViewModel.cs) previously used a UI-thread loop that could keep consuming queued samples in one dispatcher turn.

Risk:
- Under bursty telemetry or overlapping producers, one long dispatcher turn can starve layout/render/input and create frame collapse.

### Binding and PropertyChanged storms
- [src/OmenCoreApp/ViewModels/DashboardViewModel.cs](../src/OmenCoreApp/ViewModels/DashboardViewModel.cs) raises a large set of bound properties per sample.
- [src/OmenCoreApp/ViewModels/GeneralViewModel.cs](../src/OmenCoreApp/ViewModels/GeneralViewModel.cs) accepted every sample and projected directly to many bound properties.

Risk:
- Over-reactive projection causes excessive invalidation and compositing pressure, especially when combined with chart/history updates.

### Polling and timer overlap
- Monitoring, tray, OSD, uptime, and other runtime timers coexist.
- Some overlap is intentional, but aggregate cadence can still produce UI churn when projection is not aggressively rate-limited.

Risk:
- Multiple independent producers can synchronize into repeated UI-thread spikes.

### Runtime/UI coupling
- Telemetry is correctly collected in service layers, but UI projection paths were still too eager in key surfaces.

Risk:
- Telemetry cadence and rendering cadence were not sufficiently decoupled.

### Shared runtime fanout amplification
- `RuntimeStateEngine.StateChanged` was still being used as a transport for monitoring samples in addition to mode/projection state.

Risk:
- Ordinary telemetry timestamps and small sample noise triggered subscribers that should only react to mode, fan, curve, and linked-state changes.
- This multiplied downstream tray and popup work without representing a meaningful control-state transition.

## Root Cause Analysis

Primary root cause:
- UI projection was too reactive relative to incoming telemetry cadence.

Contributing causes:
- Dashboard sample projection processed queued samples in a potentially unbounded UI-thread turn.
- Dashboard and General surfaces projected minor telemetry noise with high PropertyChanged density.
- Multiple timer/polling sources increased probability of overlapping projection work.

Outcome in the field:
- UI queue saturation and render starvation, perceived as severe frame collapse (0.3 FPS class behavior).

## Mandatory Architecture Direction

Telemetry cadence and rendering cadence must remain decoupled:
- telemetry can stay responsive internally
- UI projection must be throttled, coalesced, stabilized, and rate-limited
- UI must not redraw on every sample

Target UX:
- quiet, stable control-surface behavior
- predictable frame pacing
- low redraw frequency at idle
- smooth interactions during monitoring load

## Changes Implemented In This Audit Pass

### Dashboard projection decoupling
Updated [src/OmenCoreApp/ViewModels/DashboardViewModel.cs](../src/OmenCoreApp/ViewModels/DashboardViewModel.cs):
- Added minimum UI projection interval gate
- Added significant-delta gate for temp/load/power/RPM
- Replaced unbounded per-dispatch sample loop with one-sample-per-dispatch scheduling
- Preserved coalescing while forcing yield back to dispatcher between projections

Effect:
- Reduces dispatcher monopolization risk
- Reduces binding/layout storms from minor telemetry jitter

### General projection throttling
Updated [src/OmenCoreApp/ViewModels/GeneralViewModel.cs](../src/OmenCoreApp/ViewModels/GeneralViewModel.cs):
- Added minimum projection interval gate
- Added significant-delta and telemetry-state-change gating
- Suppresses minor rapid sample noise

Effect:
- Prevents telemetry oscilloscope behavior on General tab
- Reduces unnecessary PropertyChanged churn

### Runtime counter instrumentation
Added shared runtime UI metrics collection in [src/OmenCoreApp/Services/RuntimeUiPerformanceCounters.cs](../src/OmenCoreApp/Services/RuntimeUiPerformanceCounters.cs) and instrumented:
- Main dispatcher-post/invoke and monitoring-sample coalescing/projection paths in [src/OmenCoreApp/ViewModels/MainViewModel.cs](../src/OmenCoreApp/ViewModels/MainViewModel.cs)
- Dashboard telemetry receive/project/skip/requeue/dispatcher-post paths in [src/OmenCoreApp/ViewModels/DashboardViewModel.cs](../src/OmenCoreApp/ViewModels/DashboardViewModel.cs)
- General telemetry receive/project/skip paths in [src/OmenCoreApp/ViewModels/GeneralViewModel.cs](../src/OmenCoreApp/ViewModels/GeneralViewModel.cs)

Exposed these counters in support bundles via [src/OmenCoreApp/Services/Diagnostics/DiagnosticExportService.cs](../src/OmenCoreApp/Services/Diagnostics/DiagnosticExportService.cs) under the "UI Dispatcher + Projection Counters" section.

Additional ratios now exported:
- `TotalProjectedSamples`
- `ProjectionAmplificationRatio`
- `DispatcherAmplificationRatio`
- `DashboardProjectionAcceptanceRatio`
- `GeneralProjectionAcceptanceRatio`
- `MainProjectionAcceptanceRatio`

These ratios make it possible to distinguish between raw sample intake, accepted projection work, and dispatcher pressure in field bundles instead of only looking at absolute counters.

### Runtime state engine telemetry decoupling
Updated [src/OmenCoreApp/Services/RuntimeStateEngine.cs](../src/OmenCoreApp/Services/RuntimeStateEngine.cs) and [src/OmenCoreApp/App.xaml.cs](../src/OmenCoreApp/App.xaml.cs):
- Removed monitoring sample storage and publication from `RuntimeStateEngine`
- Kept runtime-state snapshots focused on performance mode, fan preset, curve name, and linked-state projection
- Routed tray telemetry updates directly from `MainViewModel.LatestMonitoringSample`

Effect:
- Removes telemetry-triggered fanout from a shared runtime authority event
- Keeps `StateChanged` aligned with actual control-state transitions instead of sample cadence

### Main summary delta gating
Updated [src/OmenCoreApp/ViewModels/MainViewModel.cs](../src/OmenCoreApp/ViewModels/MainViewModel.cs):
- Added old/new rendered summary comparison for CPU, GPU, memory, storage, and CPU clock text
- `PropertyChanged` for summary bindings now only fires when the displayed text actually changes

Effect:
- Suppresses binding churn from minor telemetry noise
- Reduces redraw pressure on the primary surface without sacrificing sample freshness for consumers that truly need the full sample

### Hidden General surface suppression
Updated [src/OmenCoreApp/ViewModels/GeneralViewModel.cs](../src/OmenCoreApp/ViewModels/GeneralViewModel.cs) and [src/OmenCoreApp/Views/GeneralView.xaml.cs](../src/OmenCoreApp/Views/GeneralView.xaml.cs):
- Added a visibility-controlled telemetry projection gate for the General surface
- Disabled `GeneralViewModel` telemetry projection while the General view is hidden or unloaded

Effect:
- Removes steady-state `PropertyChanged` churn for an inactive tab
- Preserves projection when the surface is actually on screen instead of paying that cost continuously in the background

### Hidden dashboard dormancy
Updated [src/OmenCoreApp/ViewModels/DashboardViewModel.cs](../src/OmenCoreApp/ViewModels/DashboardViewModel.cs) and [src/OmenCoreApp/Controls/HardwareMonitoringDashboard.xaml.cs](../src/OmenCoreApp/Controls/HardwareMonitoringDashboard.xaml.cs):
- Added an explicit dashboard telemetry projection gate controlled by actual surface visibility and minimized state
- Hidden/minimized dashboard state now keeps only the latest queued sample instead of continuously mutating history collections
- Resume from dormancy schedules one coalesced projection instead of replaying hidden-time graph churn

Effect:
- Stops hidden dashboard graph/history mutation
- Prevents background dispatcher work and collection churn from an inactive monitoring surface
- Aligns dashboard cost with actual visibility instead of ongoing telemetry cadence

### Tray and popup render-state quieting
Updated [src/OmenCoreApp/Utils/TrayIconService.cs](../src/OmenCoreApp/Utils/TrayIconService.cs) and [src/OmenCoreApp/Views/QuickPopupWindow.xaml.cs](../src/OmenCoreApp/Views/QuickPopupWindow.xaml.cs):
- Tray refresh now caches the last rendered tooltip/menu/icon state and skips identical output on timer ticks
- Temperature badge regeneration is now bounded by visible badge-temperature changes instead of every tray refresh interval
- Quick popup telemetry refresh now caches the last rendered text state and skips no-op text rewrites while visible

Effect:
- Reduces tooltip/menu mutation pressure on the tray timer
- Lowers popup redraw churn without changing monitoring accuracy
- Keeps fixed refresh timers cheap when rendered telemetry is stable
