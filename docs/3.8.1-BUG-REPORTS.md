## BUG-3810-004 - GitHub #144 OMEN 17 `8A18` Temperature Freeze And Performance Fan Lock

**Reporter environment**

- OMEN by HP Laptop 17-ck1xxx, ProductId `8A18`, SystemSku `5CD228235Q`.
- Intel Core i7-12700H and NVIDIA RTX 3080 Ti Laptop GPU.
- WMI thermal policy V1, two fans, WMI BIOS control selected, Secure Boot enabled.
- In v3.8.0, `8A18` is not an exact capability entry and resolves through broad OMEN 17 family defaults.

### Submitted Log Evidence

- The worker initially reports live CPU package/core and NVIDIA GPU temperatures; a duplicate worker exits cleanly because the mutex is held.
- Runtime authority moves from WMI BIOS to ACPI and then to the worker-backed fallback for this model.
- CPU and GPU readings repeatedly trigger frozen-sensor warnings; two UI-thread sensor reads exceed the 250 ms timeout.
- A Max verification expects about 6000 RPM but repeatedly reads 3300 RPM and fails after four attempts.
- Applying the combined Performance profile preserves WMI `Default` policy, enables the built-in four-point Performance curve, and immediately commands fan levels from software. At 81C it requests Max.
- Command-success messages therefore do not establish stable physical fan response or reliable temperature authority.

### Investigation And Safety Boundary

1. Keep the new exact `8A18` identity conservative; do not copy the `17-ck2` EC profile.
2. Reproduce OEM Auto, power-only Performance, combined Performance, and Max as separate cases while recording both fan zones and an independent temperature/RPM source.
3. Determine whether the reported "lock" is software curve ownership, firmware reclaim, invalid RPM scaling, or stale telemetry before changing fan policy.
4. Preserve Restore OEM Auto as the escape path and avoid direct EC writes while Secure Boot/PawnIO and register safety are unverified.

### Implemented Source Fixes

- Exact `8A18` identity now retains WMI V1 control while disabling unverified direct EC, independent curves, and physical RPM claims.
- `GetFanLevel()` fallback values are marked `RpmSource.Estimated`; `level × 100` remains available as an explicitly labeled UI estimate only.
- Verification records RPM provenance, rejects estimated values as tachometer evidence, and uses matching fan-level readback without double-counting the same value as RPM.
- Enhanced verification no longer rewrites a successful firmware call as failed merely because physical confirmation is unavailable.
- Worker-backed model authority remains labeled `LHM Worker Override`, removing the immediate false authority switch seen in the submitted log.
- Fan Control, guided diagnostics, and command summaries visibly label estimated RPM.
- Regression coverage was added for exact model safety, V1 estimated source, physical-versus-estimated verification, mismatch wording, worker authority identity, and telemetry display.

### Required Acceptance

- Thirty-minute bounded load test with no unexplained frozen temperature authority.
- Requested fan state, WMI policy/readback, and both physical fan RPMs agree or are explicitly marked unverified.
- Switching Performance -> Balanced/OEM Auto promptly releases software curve ownership.
- Max verification uses model-correct RPM expectations before it can report failure or success.

## BUG-3810-001 - GitHub #141 OMEN 16-ap0xxx Dedicated Key Missing; Fn+F2 False-Triggers OmenCore

**Reporter environment**

- HP OMEN Gaming Laptop 16-ap0xxx, 2025 AMD/NVIDIA variant.
- Capability identity: OMEN 16 (2025) ap0xxx AMD.
- Capability ProductId `8D26`; keyboard ProductId `8D24`.
- System SKU `5CD552J0CY`.
- AMD Ryzen AI 7 350, Radeon 860M, RTX 5070 Laptop, 32 GB RAM.
- Windows 11 Pro 25H2, build `26200.8655`.
- OmenCore `3.8.0.0`, running as Administrator.

### Reported Behavior

- The dedicated OMEN key produces no captured event and no log entry.
- Fn+F2 lowers brightness and also toggles the OmenCore window.
- The false trigger is logged as `VK=0xFF, Scan=0x002B`.
- `strictOmenKeyMode: true` does not prevent the behavior in the reporter's binary.
- Physical Fn+P does not cycle performance modes, although the software `Ctrl+Shift+P` hotkey works.
- Relevant settings were enabled: OMEN interception, strict mode, and firmware Fn+P cycling.

### Expected Behavior

- Fn+F2 changes brightness only and is never consumed or treated as an OMEN key.
- The dedicated OMEN key performs the configured OMEN-key action exactly once.
- Fn+P cycles the performance profile exactly once when this model exposes a verified firmware event; otherwise diagnostics explicitly report that no supported event was observed.

### Current Source Audit

The intake baseline already contains safeguards for the reported false-trigger signature:

- `OmenKeyService.BrightnessConflictScanCodes` contains `0x002B`.
- `IsOmenKey` rejects `VK=0xFF / Scan=0x002B` in strict mode because `0x002B` is not an allowed OMEN scan.
- `HotkeyAndMonitoringTests.IsOmenKey_RejectsOemOmenVkWithBrightnessFalsePositiveScan` tests the exact reported pair.

This conflicts with the 3.8.0 field log, which says the pair was accepted. Treat this as a release-artifact/provenance or runtime-path discrepancy until proven otherwise. Do not merely add another copy of the same unit test and declare the issue fixed.

The dedicated key may use the HP WMI/ACPI event path rather than a keyboard-hook event on this model. Current startup behavior listens only for the narrow Fn+P WMI event when the keyboard hook is active. That avoids historical duplicate/brightness triggers, but it can also omit a genuine WMI-only OMEN launch event. This is a hypothesis, not yet confirmed for `8D26`.

### Investigation Order

1. Verify the reporter's executable provenance: ZIP vs installer, file version, SHA256, release asset name, and whether an older executable survived an upgrade.
2. Reproduce the exact classifier call in a Release build and confirm `VK=0xFF / Scan=0x002B` returns false.
3. Capture raw key-hook events for dedicated OMEN, Fn+F2, and Fn+P without changing the allow-list.
4. Capture `hpqBEvnt` properties for the same three actions, including `eventId`, `eventData`, class, timestamp, and whether the keyboard hook was active.
5. Determine whether `8D26` needs a model-scoped WMI launch mapping, a different event code, or only a packaging correction.
6. Confirm that a recent never-intercept brightness event suppresses any coincident WMI launch event without suppressing a later real OMEN-key event.

### Candidate Code Areas

- `src/OmenCoreApp/Services/OmenKeyService.cs`
- `src/OmenCoreApp.Tests/Services/OmenKeyServiceTests.cs`
- `src/OmenCoreApp.Tests/Services/HotkeyAndMonitoringTests.cs`
- `src/OmenCoreApp/Services/Diagnostics/DiagnosticExportService.cs`
- `src/OmenCoreApp/Controls/DiagnosticExportControl.xaml.cs`
- Model-scoped capability data only if the captured event proves a model distinction is necessary.

### Required Automated Coverage

- Release-build test for `VK=0xFF / Scan=0x002B` with strict mode both enabled and disabled: it must always be rejected.
- Hook/WMI arbitration tests for a WMI-only dedicated OMEN event while the hook is active.
- Deduplication test where one physical OMEN press appears on more than one input path.
- Brightness-never-intercept suppression test followed by a valid OMEN event outside the guard window.
- Fn+P test based on the captured event, not an assumed global code.
- Diagnostic snapshot test that records active watchers, selected query/event codes, and the last accepted/rejected candidate reason.

### Hardware Acceptance Criteria

- Ten Fn+F2 presses change brightness and produce zero OmenCore actions.
- Ten dedicated OMEN-key presses produce ten actions, with no double fire.
- Ten Fn+P presses produce the documented profile sequence if supported on `8D26`.
- Other function keys, Print Screen, Scroll Lock, Pause, and Num Lock remain unaffected.
- Exported diagnostics identify which input path accepted each valid action and why rejected candidates were rejected.

## BUG-3810-002 - GitHub #143 Victus 15 `8DCD` Fan Drops Below 2000 RPM Above 80C

**Reporter environment**

- WMI model: `Victus by HP Gaming Laptop 15-fa2xxx`.
- Exact ProductId / baseboard ProductId: `8DCD`.
- System SKU / HP support product number: `5CD533BB9L`.
- Current capability source: exact ProductId, medium confidence, not user-verified.
- Keyboard remains unknown.
- Issue title reports Performance mode falling from about 5000 RPM to below 2000 RPM while CPU temperature is above 80C; the low speed remains until the fan mode is changed manually, with recovery after roughly five minutes.

### Severity And Immediate Field Guidance

Treat this as thermal-safety critical until disproven. If cooling collapses while temperature rises, stop the workload and use Restore OEM Auto or another known-good OEM control path. Do not continue a stress test merely to obtain a longer log.

### Relationship To 3.8.0

ProductId `8DCD` was added for GitHub #138 in 3.8.0. The exact profile is deliberately conservative:

- WMI fan/profile control enabled;
- direct EC fan writes disabled;
- independent fan curves disabled;
- WMI thermal-policy fallback enabled;
- no guessed PL1/PL2 override.

Issue #143 is therefore new field evidence against the same model policy. It may indicate firmware reclaim, a Performance-to-Auto handoff, a stale WMI floor, command/readback mismatch, monitoring error, or another actor taking fan ownership. The issue body does not yet distinguish those cases.

### Evidence Required Before Changing Policy

- OmenCore version and artifact SHA256.
- BIOS version, Windows build, CPU/GPU identifiers, and whether OGH/HP services are installed or running.
- Full support bundle covering startup through the first drop and safe recovery.
- Exact timestamps for: workload start, Performance apply, 5000 RPM observation, drop below 2000 RPM, temperature crossing 80C, manual mode change, and recovery.
- `core-control-readiness.txt`, `wmi-command-history.txt`, `tuning-fan-focus.txt`, `launch-readiness.txt`, model identity, and main application log.
- CPU and GPU fan RPM separately; requested mode/level; WMI policy; duty/level readback; temperature; package power; and ownership/reclaim indicators.
- Reproduction with OGH fully closed, then a separate control run using OEM behavior only. Do not run both controllers concurrently.
- Whether Restore OEM Auto recovers immediately, whether Max recovers immediately, and whether only another profile recovers after five minutes.

### Investigation Order

1. Confirm telemetry is real: compare OmenCore RPM with an independent readback if available and inspect both fan zones.
2. Correlate the first RPM collapse with OmenCore's command history and thermal-policy changes.
3. Check for external ownership: OGH, HP services, BIOS policy reclaim, sleep/resume, game profile automation, or startup restore.
4. Check whether 3.8.0's `8DCD` exact profile changed behavior relative to 3.7.1 family fallback.
5. Reproduce safe transitions: OEM Auto -> Performance, Performance hold, Performance -> OEM Auto, and Max where supported.
6. Only after readback identifies the failing path, choose between a model-specific WMI handoff change, hold/reassert logic, capability reduction, or diagnostics-only warning.

- A sustained high-temperature / unexpectedly-low-RPM state emits an actionable warning and never reports the requested mode as verified solely from command success.

## BUG-3810-003 - Saved Custom Fan Curve Is Not Initially Selected

**Source:** Attached field log `OmenCore_20260618_165624.log`  
**Observed version:** v3.8.0 portable build  
**Initial severity:** Medium-High because a misleading selection can make the user believe a different cooling policy is active.  
**3.8.1 state:** Implemented. `ResolveInitialPresetSelection()` (`src/OmenCoreApp/ViewModels/FanControlViewModel.cs`) now checks for a valid ad-hoc `CustomFanCurve` whenever `LastFanPresetName` is empty or does not resolve to an existing preset, migrates the UI selection to the materialized `Custom` preset, and repairs the saved preset name via `RepairLastPresetSelection()` — without calling `FanService.ApplyPreset` during construction. Covered by `FanControlViewModelTests.Constructor_MigratesSavedAdHocCurveWhenLastPresetIsMissingOrStale` (null/empty/renamed cases). The original attached log's exact config file was never obtained, so this fix addresses the general missing/stale-name class of bug described in the log rather than a confirmed reproduction of that specific session.

### Log Evidence

The startup log shows:

```text
Loaded 3 custom fan presets from config
Loaded last applied custom fan curve from config (15 points)
Startup fan restore is disabled - skipping automatic fan preset reapply
```

The machine exposes WMI fan control, fan curves, one model-database fan zone, fan-level readback, and no reliable RPM readback. The log proves that the saved curve data loads, but it does not record `LastFanPresetName` or a final initial UI selection. No fan preset is reapplied because startup fan restore is disabled.

### Current Source Audit

`FanControlViewModel` deliberately separates initial UI hydration from hardware application:

- custom presets are loaded from `AppConfig.FanPresets`;
- the ad-hoc saved curve from `AppConfig.CustomFanCurve` is restored as a `Custom` manual preset when needed;
- `_suppressApplyOnSelection` prevents constructor-time selection from writing hardware;
- `ResolveInitialPresetSelection()` selects `LastFanPresetName` when it resolves and otherwise falls back to `Auto`;
- `MainViewModel.RestoreSavedSettingsAsync()` separately controls hardware reapplication through `StartupRestoreFansEnabled` and model safety policy.

This means a valid saved custom curve can be present while Auto is highlighted if `LastFanPresetName` is empty, stale, renamed, deleted, or not persisted when the ad-hoc Custom curve is applied. The attached log is consistent with that path, but the config file is needed to identify which condition occurred.

### Intended Behavior

- The initially selected card/editor must represent the last user-facing fan preset or custom curve saved in configuration.
- If a valid saved Custom curve was the last selection, Custom is highlighted and its points appear in the editor.
- Initial UI selection must not write fan hardware.
- Startup hardware reapplication remains controlled only by the explicit startup fan-restore setting and its model safety gate.
- When startup restore is disabled, the UI must distinguish `Saved selection: Custom` from `Active hardware state: OEM/unknown` instead of implying the saved curve is currently controlling fans.

### Evidence Needed

- Redacted config fields: `LastFanPresetName`, `CustomFanCurve`, `FanPresets`, `StartupRestoreFansEnabled`, and `AllowStartupRestoreOnOmen16OrVictus`.
- Screenshot of the Fan Control page immediately after launch, before clicking a card.
- Diagnostic values for selected UI preset, `_fanService.ActivePresetName`, current backend mode/readback, curve-active state, and startup restore decision.
- Whether the curve was applied through Apply Curve, Save Preset, Quick Access, a game profile, or a previous startup restore.

### Candidate Code Areas

- `src/OmenCoreApp/ViewModels/FanControlViewModel.cs`
- `src/OmenCoreApp/ViewModels/MainViewModel.cs`
- `src/OmenCoreApp/Services/FanService.cs`
- `src/OmenCoreApp/Models/AppConfig.cs`
- `src/OmenCoreApp/Services/Diagnostics/DiagnosticExportService.cs`
- Fan-control view-model, startup-restore, and configuration-persistence tests.

### Required Automated Coverage

- A valid `CustomFanCurve` plus `LastFanPresetName = "Custom"` initially selects Custom without calling `FanService.ApplyPreset`.
- A valid ad-hoc custom curve with missing/stale `LastFanPresetName` follows the agreed migration rule and does not silently discard the curve in favor of Auto.
- A renamed or deleted custom preset falls back predictably and repairs or clearly reports the stale name.
- Startup fan restore disabled: selection hydrates, zero hardware writes occur, and active state is not falsely reported as Custom.
- Startup fan restore enabled and allowed: saved Custom is selected and applied once, with UI and confirmed runtime state synchronized.
- Models without curve support continue to select Auto/OEM behavior and never expose a saved manual curve.

### Acceptance Criteria

- Relaunching after saving/applying Custom shows Custom and the exact saved points.
- Relaunching with startup fan restore disabled performs no fan write and labels the runtime state honestly.
- Relaunching with safe startup restore enabled applies at most once and reports command/readback result.
- Auto, Direct, built-in presets, named custom presets, deleted presets, and unsupported-model fallbacks retain correct initial selection.

## PERF-3810-001 - Background Memory And Responsiveness Pass

**Source:** Multiple user reports of 400+ MB RAM while OmenCore is in the background.  
**Status:** Measurement and optimization planned for v3.8.1; full scenario-matrix measurement still requires physical OMEN/Victus hardware and has not been done.  
**Scope:** Windows WPF process plus `OmenCore.HardwareWorker`; Linux/Avalonia regressions should still be checked where shared services change.  
**3.8.1 partial progress:** Audited `BackgroundTimerRegistry` against actual recurring loops in the codebase and registered three that were previously invisible to diagnostics: the global `HpWmiBios` heartbeat (`Critical`), the `WmiFanController` countdown-extension reassert loop (`Critical`, directly relevant to BUG-3810-002/#143), and the optional `RazerService` Chroma heartbeat (`Optional`). This does not change memory/CPU behavior; it only makes existing loops visible in `core-control-readiness.txt` and other exports so the scenario-matrix measurement (still pending hardware) has accurate timer inventory to work from.

### Problem Definition

Task Manager's Memory column, process working set, private bytes, managed heap, and combined main/worker footprint are different measurements. A useful 3.8.1 pass must capture all of them and must not claim a leak from one screenshot. The attached startup log contains no memory snapshot, so it corroborates active background services but does not prove the reported 400+ MB footprint.

The application already exports `runtime-performance.txt`, `runtime-performance-bounded.txt`, and `background-timers.txt`, including working set, private memory, managed heap, collection counts, threads, handles, UI dispatcher/projection counters, cadence, hidden-surface suppression, and worker processes. Use these as the common evidence format instead of adding continuous high-cost profiling.

### Scenarios To Measure

Use the same Release build, config, hardware, and 15-minute warm-up for each scenario:

1. Fresh launch with main window focused on General.
2. Monitoring/dashboard visible with charts active.
3. Main window minimized to tray, OSD off, RGB scenes off, no games tracked.
4. Tray-only with low-overhead mode enabled.
5. Quick Access popup repeatedly opened/closed.
6. OSD enabled at its normal refresh rate.
7. Fan curve actively controlling versus OEM Auto.
8. After 2 hours tray-only to detect retained collections, views, timers, handles, or worker growth.

Capture at 0, 15, 30, 60, and 120 minutes: main and worker working set/private bytes, managed heap/fragmentation, Gen 0/1/2 counts, allocation rate if a profiler is attached, CPU, threads, handles, active timers, loaded optional assemblies, monitoring cadence, and dispatcher/projection deltas.

### Provisional 3.8.1 Budgets

These are engineering targets, not claims about 3.8.0:

- Tray-only, OSD off after warm-up: combined main + worker working set below 300 MB on the reference machine.
- No ordinary background scenario should remain above 400 MB combined without diagnostics identifying an enabled high-cost feature.
- Thirty-minute tray-only growth below 25 MB and two-hour growth should plateau rather than remain monotonic.
- Tray-only average process CPU below 1% on an otherwise idle reference machine.
- Main-window restore and navigation should visibly respond within 250 ms at the 95th percentile; no multi-second UI-thread stalls.
- Quick Access should open within 150 ms at the 95th percentile after first warm-up.
- No increase in fan-control latency or missed thermal samples as polling/UI work is reduced.

If hardware providers make an absolute budget infeasible, preserve the before/after evidence and require at least a 25% reduction in the affected scenario rather than redefining success from intuition.

### Investigation Areas

- Verify only one hardware monitor/provider graph and one `HpWmiBios` heartbeat exist.
- Compare the main process and hardware worker separately; ensure a stale worker is not left behind after restart/crash.
- Audit `BackgroundTimerRegistry` against actual timers/loops and register any missing long-lived loop.
- Confirm hidden dashboard/general/chart surfaces stop projection and do not retain window/view-model subscriptions.
- Bound chart histories, fan telemetry collections, logs, diagnostic samples, RGB frames, and process/game histories.
- Defer optional RGB SDKs, tuning providers, conflict scans, charts, and device integrations until their page/feature is used.
- Review LibreHardwareMonitor lifetime, sensor enumeration caches, WMI objects/watchers, images/bitmaps, tray icon renders, and event-handler unsubscription.
- Reduce duplicate polling and dispatcher posts; coalesce latest telemetry rather than queueing stale samples.
- Keep fan control, thermal protection, watchdog behavior, and active OSD cadence independent from cosmetic UI throttling.
- Profile Release builds with Visual Studio Profiler, PerfView, or `dotnet-counters`; compare retained types before changing GC behavior.

### Required Automated And Repeatable Checks

- Lifecycle tests prove views/view-models and timers can be unloaded/disposed without retained event subscriptions.
- Collection-bound tests cover charts, dashboard history, fan telemetry, logs, and diagnostic snapshot storage.
- Tray-mode tests assert hidden surfaces do not project each monitoring sample.
- Worker ownership tests ensure one live worker and cleanup/recovery without duplicates.
- Cadence tests preserve fan/thermal safety intervals while optional UI work slows or sleeps.
- A repeatable PowerShell or diagnostic-run procedure records scenario, build SHA, configuration hash, sample times, and aggregate main/worker metrics.
- Release notes report median and peak memory, not only the best single sample.

### Acceptance Criteria

- A baseline and post-change report exists for every scenario above.
- The 400+ MB report is either reproduced and reduced or explained by a named optional feature with a verified off-state reduction.
- Memory stabilizes during the two-hour tray test; handles, threads, timers, and loaded views do not grow without bound.
- Responsiveness budgets pass without weakening fan safety, monitoring truthfulness, or command readback.

## UX-3810-001 - GPU Overclock Save And Startup Reapply Are Not Clear Or Consistent

**Reporter:** Robert (`float`) / Discord, 2026-06-19  
**Report:** "Is there a way to save the gpu overclock speeds or do I have to keep saving it when booting up"  
**Initial severity:** Medium-High; repeated manual tuning invites mistakes, while careless automatic OC restore can cause boot/login instability.  
**3.8.1 state:** Discoverability implemented; the underlying safety logic was already correct on review. Added a `Startup:` chip to the Tuning page (next to the existing Confirmed/Requested chips) that names the exact gate blocking reapply (global Startup Hardware Restore, Tuning-category restore, or the sensitive-model override) or confirms it is enabled. Also fixed a related safety gap found during review: `StartupRestorePolicy.IsSensitiveModel()` matched only the literal substring `"OMEN 16"`, which real HP WMI model strings (e.g. `"OMEN Gaming Laptop 16-ap0xxx"`) often do not contain, so the OMEN 16/Victus safety override could silently fail to engage. Now matches any `16-`/`16t-` board-pattern OMEN model plus any Victus model. AMD ADL2 now shows an explicit "manual apply only" notice in the UI since no startup-persistence path exists for it. Physical reboot validation on NVIDIA/AMD hardware is still required per the release gate below.

### Current 3.8.0 Behavior

The answer depends on GPU backend and action used:

| Path | Saves named profile | Saves current values | Reapplies at OmenCore startup |
|---|---|---|---|
| NVIDIA NVAPI, Save Profile | Yes | In `GpuOcProfiles` | No; profile save/selection alone does not authorize hardware writes |
| NVIDIA NVAPI, Apply GPU Settings | No | Yes, to `GpuOc` | No; `ApplyGpuOcValues(... persistToConfig: true)` calls `SaveGpuOcToConfig(applyOnStartup: false)` |
| NVIDIA NVAPI, Test Apply then Keep | No new named profile; confirms current values | Yes, to `GpuOc` plus confirmation metadata | Eligible; Keep sets `ApplyOnStartup=true` |
| AMD ADL2, Apply AMD GPU OC | No equivalent persistence found | No equivalent config save found | No equivalent startup reapply path found |

Even the NVIDIA confirmed path runs only when all applicable safety gates permit it:

1. `GpuOc.ApplyOnStartup` must be true, which currently comes from **Test Apply -> Keep**.
2. Settings -> startup hardware restore (`EnableStartupHardwareRestore`) must be enabled.
3. The Tuning startup category must be enabled.
4. OMEN 16/Victus-class machines also require the sensitive-model startup-restore override.
5. OmenCore must launch after login and initialize NVAPI successfully; this is application-start restore, not firmware persistence before Windows/OmenCore starts.

The current design is safety-minded but hard to discover. A user can save a named profile and reasonably assume it will return after reboot, while the application has only saved it for manual selection. Pressing ordinary Apply after previously confirming a startup profile also sets `ApplyOnStartup=false`, so the status can change without a clear explanation.

### 3.8.1 Product Decision

Keep automatic overclocking opt-in and confirmation-based. Do not make every saved profile auto-apply. Improve the workflow so users can deliberately choose one confirmed startup profile and can always see why it did or did not apply.

Recommended interaction:

- Save Profile stores reusable values only.
- Test Apply verifies stability for a bounded period.
- Keep marks those exact values as confirmed.
- A clearly labelled **Reapply this confirmed GPU profile when OmenCore starts** control enables startup restore, subject to the global/category/model safety gates.
- UI shows `Saved`, `Selected`, `Confirmed`, `Startup enabled/blocked`, and `Applied + verified` as separate states.
- Changing values after confirmation invalidates or clearly preserves the previous confirmed startup profile; it must never silently imply the new untested values will boot-apply.
- AMD either receives the same confirmed persistence contract with ADL2 readback/recovery or is explicitly labelled `manual apply only` for 3.8.1.

### Candidate Code Areas

- `src/OmenCoreApp/ViewModels/SystemControlViewModel.cs`
- `src/OmenCoreApp/Views/TuningView.xaml`
- `src/OmenCoreApp/Models/AppConfig.cs`
- `src/OmenCoreApp/Models/StartupRestorePolicy.cs`
- `src/OmenCoreApp/Models/TuningStartupRecoveryGuard.cs`
- `src/OmenCoreApp/Models/TuningStartupRecoveryCoordinator.cs`
- `src/OmenCoreApp/Hardware/NvapiService.cs`
- AMD ADL2 service and models if AMD persistence is implemented.
- `src/OmenCoreApp/Services/Diagnostics/DiagnosticExportService.cs`

### Diagnostics Required

- GPU vendor/backend and detected tuning capability.
- Selected named profile and saved custom profile count.
- Requested, confirmed, and live/read-back core, memory, power, and voltage values.
- `GpuOc.ApplyOnStartup`, global restore, Tuning category, sensitive-model override, and final effective allow/block reason.
- Last startup restore attempt time, result per value, guardrail/clamp changes, and conflict detection result.
- Whether a safe reset occurred because a Test Apply was interrupted or never confirmed.

### Required Automated Coverage

- Save Profile persists reusable values but performs no hardware write and does not opt into startup restore.
- Test Apply -> Keep persists exact confirmed values, clears pending state, and enables eligible startup reapply.
- Test timeout/cancel/crash restores safe previous values and cannot enable startup reapply.
- Ordinary Apply has an explicit tested behavior for an existing confirmed startup profile; no silent or ambiguous flag transition.
- Startup restore is blocked independently by global, Tuning-category, and sensitive-model gates with a user-visible reason.
- Successful NVIDIA startup restore applies once after backend readiness and verifies every supported value.
- Partial NVAPI writes report partial state and do not claim the complete profile is active.
- Driver-locked clock/power controls remain excluded rather than treated as successful zero/default writes.
- AMD persistence either has equivalent save/confirm/restore/recovery tests or the UI/tests enforce manual-only wording.
- A conflicting tuner such as MSI Afterburner or NVIDIA automatic tuning blocks or explicitly confirms ownership before restore.

### Hardware Acceptance Criteria

- On a supported NVIDIA system, create a conservative profile, Test Apply, Keep, enable startup restore, reboot, launch OmenCore, and confirm requested/read-back offsets match exactly once.
- Repeat with global restore off, Tuning restore off, and the sensitive-model gate off; each must perform zero OC writes and state the blocking reason.
- Confirm Reset to Defaults disables startup OC and restores zero/default values.
- Confirm an unconfirmed or interrupted test safely resets on next launch.
- Test sleep/resume and a GPU driver restart separately; document whether 3.8.1 reapplies, warns, or intentionally requires manual action.
- On AMD hardware, validate the implemented equivalent or confirm the UI states that startup persistence is unavailable.

The current safe path is Test Apply, then Keep, and enabling Startup Hardware Restore plus the Tuning category in Settings. OMEN 16/Victus also has an extra safety override. AMD automatic restore is not currently implemented. We are tracking the confusing/inconsistent workflow for 3.8.1.

## MODEL-3810-002 - GitHub #145 OMEN Slim 16 `8D40` Missing Identity; Battery Care Fails; Performance Profile Resets

### Reporter Environment

- Exact ProductId / baseboard ProductId: `8D40`.

1. **Battery Care:** Enable Charge Limit fails in OmenCore (works in OMEN Gaming Hub).
2. **Performance Profile:** Always resets to Balanced when exiting and relaunching the app.

**Battery Care (not yet fixed — needs hardware evidence).** `SetBatteryCareMode`/`GetBatteryCareMode` (`HpWmiBios.cs`) are not gated by `ModelCapabilities` or any per-model database entry at all — the Settings UI always shows the toggle and always attempts the WMI call regardless of model identity. Per this tracker's standing rule, this is not guess-fixed. ## BUG-3810-005 - Discord Reports: Fan Spikes To ~5000 RPM "For No Reason" At Idle Temp

**Source:** Discord, 2026-06-23 — mattieswza [LFC] ("my fan randomly went to max speed for no reason"); Mr.Carrot (Victus 16-s0xxx) ("Common problem... Still no fix"); OsamaBiden (OMEN 16-xd0010ax) ("Agreed. It sometimes goes to 5000 rpm for no reason then drops back to idle RPM at idle temp").

### Reporter Environment

- At least two distinct models reported: Victus 16-s0xxx and OMEN 16-xd0010ax (`8BCD` family).
- No diagnostic export attached yet — this entry is triage from the symptom description and source code review, not a confirmed root cause.
- Described as a recurring/"common problem" across multiple users, which points toward shared software behavior rather than one user's hardware fault.

### Investigation And Suspected Mechanism

`FanService.CheckThermalProtection()` has three tiers:

1. **Emergency** (`maxTemp >= 95C`): activates immediately on a single sample, by design, with no debounce. The existing test `ThermalProtection_EmergencyThreshold_RaisesImmediately` explicitly encodes this as the required, intentional behavior — comment: "Emergency... immediate max fans (no debounce, safety critical)."
2. **Warning** (`maxTemp >= _thermalProtectionThreshold`, default 90C): requires 5s of sustained reading before activating (`ThermalActivateDebounceSeconds`).
3. **Release**: requires the temperature to stay below `threshold - 10C` hysteresis for 15s (`ThermalReleaseDebounceSeconds`) before restoring the prior fan state.

`cpuTemp`/`gpuTemp` are passed into this method as raw, unsmoothed per-poll readings with no rolling average or outlier rejection beyond a `[-10C, 150C]` sanity clamp. The reported symptom — fans jump to ~5000 RPM at idle temperature, then drop back to idle RPM after a delay — is consistent with a **single transient or erroneous temperature sample reaching the 95C emergency tier**, which forces 100% fans immediately (tier 1), followed by the temperature reading idle again on the next sample, which then needs the 15s release hysteresis/debounce (tier 3) to elapse before fans drop back down. That elapsed-then-released pattern matches "goes to 5000 RPM... then drops back to idle RPM at idle temp" closely.

### Why This Was Not Guess-Fixed

The zero-debounce emergency activation is a **deliberate, already-tested safety design**, not an oversight — weakening it (e.g. requiring a second consecutive sample before forcing Max) would directly contradict an existing test that encodes the no-debounce requirement as correct, and would slow the response to a genuine thermal emergency by one polling interval. Given this project's standing rule to treat thermal-safety code changes as critical and evidence-gated, this debounce was **intentionally left unchanged** in this pass. The actual question — is this a real (if very brief) thermal event, or a sensor/EC read glitch reporting a wrong value — cannot be answered without seeing the actual cpu/gpu values at the moment of an activation.

### Implemented (Diagnostics Only, No Behavior Change)

- `CheckThermalProtection` now calls `RecordFanCommand(...)` on every Emergency activation, Warning activation, and Release, including the **individual** `cpuTemp` and `gpuTemp` values (not just the combined `maxTemp` that was already logged). These entries are queryable via `FanService.GetCommandHistorySnapshot()` and already flow into existing diagnostic exports (`core-control-readiness.txt` / fan command history) without any new export plumbing.
- This makes the next occurrence diagnosable: if `core-control-readiness.txt` shows an Emergency activation with, say, `cpu=96.0C gpu=41.0C` immediately preceded by idle-range readings and immediately followed by idle-range readings again, that is strong evidence for a single-sample glitch rather than a real thermal event — at which point a targeted fix (e.g. sensor-source filtering, or a very short same-tier confirmation window) could be considered with actual evidence instead of a guess.
- Added `ThermalProtectionTests.ThermalProtection_EmergencyActivation_RecordsBothIndividualReadings`. No existing test changed; all prior thermal-protection tests (including the no-debounce requirement) still pass unmodified.

### Evidence Needed Before Any Behavior Change

- A diagnostic export (`core-control-readiness.txt` / fan command history) captured at or shortly after a spike, showing the `ThermalProtection.Emergency` entry's individual cpu/gpu values and the samples immediately before/after it.
- Whether the spike correlates with a specific monitoring source (WMI vs worker-backed LibreHardwareMonitor) — i.e. is one sensor path more prone to this than another.
- Whether the spike correlates with any system event (resume from sleep, GPU driver activity, a brief real CPU boost under a benchmark/game launch) rather than being purely random.
- Confirmation of OmenCore version/build and whether `8BCD`'s already-documented degraded-WMI-control behavior (Linux #137 territory, but these are Windows reports) plays a role.

### Candidate Code Areas (For A Future, Evidence-Backed Fix)

- `src/OmenCoreApp/Services/FanService.cs` (`CheckThermalProtection`, done for diagnostics; activation timing intentionally untouched)
- `src/OmenCoreApp/Hardware/ThermalSensorProvider.cs` and worker-backed sensor paths, if evidence points to a specific sensor source producing the bad sample
- `src/OmenCoreApp.Tests/Services/ThermalProtectionTests.cs`

### Hardware Acceptance Criteria

- Not applicable yet — this item is in the evidence-gathering stage. No fix should be attempted until a diagnostic export from an affected user is reviewed.
