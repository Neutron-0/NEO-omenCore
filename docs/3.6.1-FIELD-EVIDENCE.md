### O. High CPU while custom fan curve/hold is active
Evidence:
- Task Manager screenshot from 2026-05-13 shows OmenCore consuming ~7.2% CPU while the system is otherwise around 13% total CPU.
- `message(1).txt` shows repeated same-target WMI fan writes while a preset curve is active, for example `Fan levels set: CPU=31, GPU=31`, then later repeated low-duty writes such as 23% and 11%.
- The same log shows `[FanKeepalive] Fan level keepalive active: 20% (write #4851)` and later write #4901, confirming thousands of backend keepalive writes during one session.
- The log also shows a post-apply curve verification kick taking roughly 35 seconds with diagnostic-mode entry, repeated fan verification attempts, diagnostic-mode exit, and curve reapply.
- The log includes recurring `[WmiBiosMonitor] CPU temp fallback timed out after 500ms` entries, indicating fallback sensor contention/stall during the same high-CPU session.

Impact:
- Stable custom curves can generate continuous WMI command pressure even when the target fan percentage has not changed.
- Automatic curve verification can briefly amplify CPU/WMI load immediately after applying a preset, especially when firmware readback is slow or unresponsive.

Patch targets:
- src/OmenCoreApp/Hardware/WmiFanController.cs
- src/OmenCoreApp/ViewModels/FanControlViewModel.cs
- src/OmenCoreApp.Tests/Hardware/WmiV2VerificationTests.cs
- src/OmenCoreApp.Tests/ViewModels/FanControlViewModelTests.cs

Status:
- Remediated in current v3.6.1 slice: WMI countdown keepalive now uses a bounded cadence, skipped keepalive ticks no longer call `ExtendFanCountdown()` (which itself reissues fan-level WMI writes), low-duty manual curve hold reapply is throttled to 15s, high-duty hold retains a 5s safety cadence, preset-mode keepalive is throttled to 30s, remaining countdown-extension writes are throttled to 15s, normal manual curve writes no longer take an extra RPM snapshot for command history, automatic post-apply curve verification is limited to one fan with a 15s timeout and 10-minute same-target cooldown, and repeated CPU-temperature fallback timeouts now use capped backoff up to 300s.

### L. Fundamentals-first baseline and Linux multiplex guidance
Evidence:
- Maintainer direction in Discord: prioritize fundamentals over peripheral polish for 3.6.1, specifically fan modes and performance modes.
- Maintainer note: v3.2.5 was the last generally good baseline for fan/performance behavior, with only smaller bugs remaining.
- Historical baseline from [docs/CHANGELOG_v3.2.5.md](CHANGELOG_v3.2.5.md): fan control decoupled from performance mode by default, Quick Access fan curve support, and stable UX around performance-mode ordering.
- Historical baseline from [docs/CHANGELOG_v2.9.0.md](CHANGELOG_v2.9.0.md): tray quick-profile/fan-mode freezes had already been fixed by serialized off-UI-thread queueing.
- Discord technical note from Eric [GOG]: Linux `hp_wmi.force_multiplex=1` can restore true dGPU power-off on affected OMEN firmware and also improve performance profile detection by forcing the multiplexed WMI path.

Impact:
- Current stabilization work should treat 3.2.5 fundamentals and 2.9.0 tray responsiveness as regression anchors, not optional improvements.
- Linux documentation and diagnostics may need to call out `hp_wmi.force_multiplex=1` as a workaround when hp-wmi profile/shutdown paths are broken on newer OMEN firmware.

Patch targets:
- src/OmenCoreApp/ViewModels/MainViewModel.cs
- src/OmenCoreApp/Services/PerformanceModeService.cs
- src/OmenCoreApp/Services/FanService.cs
- src/OmenCoreApp/Views/MainWindow.xaml
- src/OmenCore.Avalonia/Services/LinuxHardwareService.cs
- docs/INSTALL.md

Status:
- Incorporated into v3.6.1 release-candidate scope: fan/performance fundamentals and tray no-freeze behavior are now regression anchors via latest-wins dispatch, service-confirmed state updates, and Linux multiplex guidance in install/diagnose documentation. Remaining Linux board graduation is post-3.6.1.

## Patch-Ready Recommendations
1. Add runtime readiness gate for undervolt capability (MSR and AMD SMU handshake) before enabling apply actions.
2. Add explicit model-confidence states (Known, Partial, Fallback) and bind them to UI capability toggles.
3. Add fan/performance consistency assertions in UI update path and tray/OSD synchronization tests.
4. Add DPI and clipping regression tests for fan cards and settings sections at 100 percent, 125 percent, 150 percent, and 200 percent scaling.
5. Add Max mode persistence verification loop to confirm post-write state and detect external overwrite.
6. Compare current fundamentals path against v3.2.5 / v2.9.0 behavior before further fan/performance rewrites so known-good decoupling and tray-queue protections are preserved.
7. Evaluate whether Linux diagnostics and install guidance should explicitly mention `hp_wmi.force_multiplex=1` for boards with broken standard hp-wmi shutdown/profile behavior.
