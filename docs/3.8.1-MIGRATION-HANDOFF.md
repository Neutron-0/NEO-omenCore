## 6. Critical Facts That Must Not Be Rediscovered Incorrectly

### #141

- The report's false key signature is exactly `VK=0xFF / Scan=0x002B`.
- Current `main` already has a test asserting that pair is rejected and already lists `0x002B` as a brightness conflict.
- The shipped 3.8.0 log says it was accepted. Verify the actual executable and SHA before editing the same guard.
- The dedicated OMEN key produces no hook log. A WMI-only event is plausible.
- Current watcher arbitration narrows to Fn+P while the keyboard hook is active, so a genuine WMI-only launch event may be omitted by design.
- Never restore a broad `SELECT * FROM hpqBEvnt` watcher; historical broad matching caused false triggers/focus stealing.

### #142

- Exact ProductId is `8E9A`; WMI model is `HyperX OMEN MAX Gaming Laptop 16t-ah100`; SKU is `1H96240HR0`.
- This is a 2026 model and is not proven equivalent to 2025 `8D41` or `8D87`.
- Existing MAX notes warn that legacy EC writes can panic/corrupt newer MAX EC state. Default to no direct EC writes.
- Exact identity and verified capability are different claims. Keep `UserVerified` false until hardware checks are complete.

### #143

- Exact ProductId `8DCD` was added in 3.8.0 because of #138.
- Its current profile intentionally disables direct EC writes and uses WMI thermal-policy fallback.
- The new issue reports a persistent fan fall below 2000 RPM from about 5000 RPM above 80C.
- A command returning success is not proof of cooling. Use RPM, level/duty readback, policy state, temperature, and timestamps together.
- Stop load testing if cooling collapses. A support bundle is not worth overheating hardware.

### Saved Custom fan selection

- The attached `OmenCore_20260618_165624.log` loads three custom presets and a 15-point saved custom curve.
- The same log says startup fan restore is disabled, so no preset should be written automatically.
- `FanControlViewModel` currently falls back to Auto when `LastFanPresetName` is missing or cannot be resolved.
- Selecting Custom in the UI and applying Custom to hardware are separate operations. Preserve that separation.
- Obtain the redacted fan-related config fields before deciding whether the bug is persistence, migration, or initial-selection resolution.

### Resources and responsiveness

- The attached log does not contain a memory snapshot and does not prove the 400+ MB reports.
- Measure OmenCore and `OmenCore.HardwareWorker` separately and together; record working set, private bytes, and managed heap.
- Existing support exports already include runtime performance, bounded samples, UI projection counters, and background timers.
- Capture the 3.8.0 baseline before making changes, then repeat identical scenarios on 3.8.1.
- Never trade fan/thermal sampling reliability for a prettier Task Manager number.

### GPU OC startup persistence

- NVIDIA named-profile save already exists, but profile save/selection does not imply startup application.
- Ordinary Apply saves current values with `ApplyOnStartup=false`.
- Test Apply -> Keep records confirmed values and sets `ApplyOnStartup=true`.
- Effective restore additionally requires the broad startup-hardware gate, Tuning category, and the OMEN 16/Victus safety override where applicable.
- Current AMD ADL2 Apply/Reset code does not have an equivalent config persistence/startup restore path.
- Treat this as a safety UX contract: never auto-apply an unconfirmed OC merely because it was saved.

## 9. Packaging And Release Gate

After all fixes, full tests, and field validation:

```powershell
.\build-installer.ps1 -Configuration Release -Runtime win-x64 -SingleFile
.\build-linux-package.ps1 -Configuration Release -Runtime linux-x64
```

Verify the generated filenames, embedded/file versions, clean-install behavior, 3.8.0 upgrade behavior, portable launch, Linux CLI launch, and SHA256 files. The tag-triggered workflow reads `VERSION.txt`, builds Windows and Linux artifacts, and creates the GitHub release.

Do not tag `v3.8.1` until:

- the full checklist in `3.8.1-BUG-REPORTS.md` is resolved;
- issue statuses in `CHANGELOG_v3.8.1.md` are evidence-backed;
- #143 has physical `8DCD` validation;
- #141 has shipped-artifact and physical key validation;
- #142 exposes only hardware-validated controls;
- saved Custom selection is correct without an unintended startup write;
- background performance has before/after scenario evidence and passes the agreed budgets;
- GPU OC startup persistence is explicit, confirmation-based, gate-aware, and physically validated on each claimed backend;
- `git status` is clean and the intended commit is pushed.
