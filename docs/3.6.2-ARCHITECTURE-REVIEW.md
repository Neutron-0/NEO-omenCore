# OmenCore 3.6.2 Architecture Review

## Summary
This review captures the stabilization-oriented architecture deltas for v3.6.2.

The core theme remains the same: reduce runtime ambiguity by keeping one authoritative owner for each state domain and letting view layers subscribe rather than invent state.

## What Changed
- Fan mode labels are now canonical at the service layer, including Gaming as a first-class runtime label.
- Built-in presets include Gaming and Quiet, which removes the need for UI-only placeholder presets.
- Hotkey routing now distinguishes Gaming from Extreme instead of collapsing them into one performance alias bucket.
- Quiet requests now select the built-in Quiet preset rather than synthesizing a manual preset.
- Extreme preset application uses an immediate path to avoid thermal ramp delay.
- Battery reads now cache the last known value and reuse it during cooldown.
- Active monitoring cadence increased from 1s to 2s to lower focused-window overhead.

## Architecture Risks Reduced
- Fewer phantom states in fan mode resolution.
- Less label drift between hotkeys, tray, OSD, and the fan service.
- Less telemetry churn when the UI is visible.
- Less chance of battery OSD flicker caused by placeholder values.

## Remaining Constraints
- MainViewModel still coordinates several runtime surfaces and remains a high-risk orchestration layer.
- Polling ownership is still distributed across multiple services and UI-adjacent flows.
- Further reductions should continue to favor runtime services and read-only subscribers.

## Validation Snapshot
- Focused syntax checks on the touched runtime files passed.
- Regression tests were added for hotkey slot resolution, canonical fan identity, curve ordering, and battery cooldown behavior.
