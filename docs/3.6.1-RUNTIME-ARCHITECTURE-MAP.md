# OmenCore v3.6.1 Runtime Architecture Map

## Purpose
This audit documents current runtime ownership and coordination paths to support architecture reduction work.

This document is intentionally focused on high-risk runtime subsystems (fan/perf, tray/OSD, hotkeys, automation, monitoring, EC access), not every feature module.

## Scope
Included in this map:
- Runtime state ownership and state mutation paths
- Polling loops and timer ownership
- EC read/write ownership
- Fan write sequencing and contention risk paths
- Dispatcher usage concentration

Out of scope for this pass:
- Non-runtime helper modules
- One-shot setup flows (installer/release scripts)
- Pure formatting/view-only code that does not mutate runtime state

## Runtime Ownership Map
### Current effective runtime state owners
1. Fan mode/preset authority:
- Primary owner: src/OmenCoreApp/Services/FanService.cs
- Canonical source fields/events: _currentFanMode, ActivePresetName, PresetApplied

2. Performance mode authority:
- Primary owner: src/OmenCoreApp/Services/PerformanceModeService.cs
- Canonical source fields/events: _currentMode, ModeApplied

3. Telemetry authority:
- Primary owner: src/OmenCoreApp/Services/HardwareMonitoringService.cs
- Canonical source fields/events: _lastSample, _samples, SampleUpdated, HealthStatusChanged

### Competing/secondary state owners (duplication risk)
1. src/OmenCoreApp/ViewModels/MainViewModel.cs
- Duplicates mutable mode state in _currentFanMode and _currentPerformanceMode
- Writes those fields from multiple paths: UI actions, tray actions, event handlers, resume checks

2. src/OmenCoreApp/Utils/TrayIconService.cs
- Maintains independent _currentFanMode and _currentPerformanceMode cache
- Can issue mode changes (FanModeChangeRequested, PerformanceModeChangeRequested)
- Can also receive mode updates (UpdateFanMode/UpdatePerformanceMode)

3. src/OmenCoreApp/Views/QuickPopupWindow.xaml.cs
- Maintains independent _currentFanMode and _currentPerformanceMode
- Emits mode change requests while also locally mutating UI state

4. src/OmenCoreApp/Views/OsdOverlayWindow.xaml.cs
- Maintains independent CurrentMode/FanMode/PerformanceMode display state

5. src/OmenCoreApp/ViewModels/GeneralViewModel.cs
- Maintains independent CurrentFanMode/CurrentPerformanceMode
- Directly applies fan + performance writes in profile commands
- Runs its own telemetry timer

6. src/OmenCoreApp/ViewModels/DashboardViewModel.cs
- Maintains independent CurrentFanMode/CurrentPerformanceMode for visualization and derived summaries

## Polling Ownership Map
### Long-running monitor loops
1. Fan monitor loop
- Owner: src/OmenCoreApp/Services/FanService.cs
- Start path: Start() -> Task.Run(MonitorLoop)
- Loop: while !token.IsCancellationRequested
- Cadence: dynamic (monitor period vs max interval) + initial delay alignment

2. Hardware monitoring loop
- Owner: src/OmenCoreApp/Services/HardwareMonitoringService.cs
- Start path: Start() -> Task.Run(MonitorLoopAsync)
- Loop: while !token.IsCancellationRequested
- Cadence: adaptive via GetEffectiveCadenceInterval(), includes pause/resume behavior

### Timer-driven polling/evaluation (core runtime)
1. Automation evaluation timer (5s)
- Owner: src/OmenCoreApp/Services/AutomationService.cs
- Timer: System.Threading.Timer _evaluationTimer
- Behavior: evaluates rules and can apply fan/performance actions

2. Hardware watchdog timer (10s)
- Owner: src/OmenCoreApp/Services/HardwareWatchdogService.cs
- Timer: System.Threading.Timer _watchdogTimer
- Behavior: monitors stale thermal updates and can force emergency fan speed

3. Tray refresh timer (2s)
- Owner: src/OmenCoreApp/Utils/TrayIconService.cs
- Timer: DispatcherTimer _updateTimer
- Behavior: periodic tray tooltip/menu refresh

4. OSD stats timer (1s) + OSD network timer (5s)
- Owner: src/OmenCoreApp/Views/OsdOverlayWindow.xaml.cs
- Timers: DispatcherTimer _updateTimer and _pingTimer
- Behavior: visible-only OSD refresh

5. OSD hotkey retry timer (2s retry cadence)
- Owner: src/OmenCoreApp/Services/OsdService.cs
- Timer: System.Threading.Timer _retryTimer

6. Quick popup refresh timer (1s)
- Owner: src/OmenCoreApp/Views/QuickPopupWindow.xaml.cs
- Timer: DispatcherTimer _updateTimer

7. General tab telemetry timer (1s)
- Owner: src/OmenCoreApp/ViewModels/GeneralViewModel.cs
- Timer: DispatcherTimer _updateTimer

8. Dashboard uptime timer (1s)
- Owner: src/OmenCoreApp/ViewModels/DashboardViewModel.cs
- Timer: DispatcherTimer _uptimeTimer

9. Memory optimizer refresh timer (2s visible / slower when inactive)
- Owner: src/OmenCoreApp/ViewModels/MemoryOptimizerViewModel.cs
- Timer: DispatcherTimer _refreshTimer

10. Settings schedule enforcement timer (30s)
- Owner: src/OmenCoreApp/ViewModels/SettingsViewModel.cs
- Timer: DispatcherTimer _scheduleTimer

11. Auto-update check timer (hours)
- Owner: src/OmenCoreApp/Services/AutoUpdateService.cs
- Timer: System.Timers.Timer _checkTimer

## Timer Ownership Map
### High-risk overlap zones
1. Thermal/fan decisions can be triggered by multiple schedulers
- FanService monitor loop
- AutomationService timer
- Tray action queue in MainViewModel
- Hotkey handlers in MainViewModel
- HardwareWatchdogService timer

2. Telemetry/view refresh has multiple independent visible timers
- HardwareMonitoringService sample stream
- Tray timer
- OSD overlay timers
- QuickPopup timer
- General tab timer
- Dashboard uptime timer

3. Resume/recovery adds delayed Task.Run + Task.Delay chains in MainViewModel and monitoring services

## EC Ownership Map
### EC backend selection and lifetime
1. Backend factory
- Owner: src/OmenCoreApp/Hardware/EcAccessFactory.cs
- Behavior: singleton IEcAccess selection (PawnIO preferred, optional WinRing0 fallback)

### Direct EC write/read owners
1. Fan EC I/O
- Owner: src/OmenCoreApp/Hardware/FanController.cs
- Operations: ReadByte fan RPM regs, WriteByte fan mode/speed/boost/restore registers

2. Power limit EC I/O
- Owner: src/OmenCoreApp/Hardware/PowerLimitController.cs
- Operations: ApplyPerformanceLimits, optional readback via ReadByte

### Service-level EC operation coordinators
1. Fan writes are serialized inside FanService
- Owner: src/OmenCoreApp/Services/FanService.cs
- Mechanism: _fanWriteLock and serialized wrappers

2. Performance mode apply has its own apply lock but separate from fan lock
- Owner: src/OmenCoreApp/Services/PerformanceModeService.cs
- Mechanism: _applyLock
- Risk: lock domains are separate; cross-domain ordering still relies on call ordering/event timing

## Fan Runtime Pipeline Map
### Ownership and sequencing
1. Primary authority
- FanService applies presets, manages curve loop, thermal protection, and emits PresetApplied

2. Write sequencing
- Serialized in FanService via _fanWriteLock wrappers
- Underlying hardware writes happen in FanController

3. Read/verify pipeline
- FanService monitor loop reads thermal provider and fan speeds each cadence
- RPM smoothing/confirmation logic is in monitor loop
- Max deferred verification has separate delayed Task.Run path

### Conflict/overwrite pressure points
1. Multiple entry points can request fan changes
- Tray actions (MainViewModel queue)
- Hotkeys (MainViewModel)
- AutomationService actions
- PowerAutomationService profile transitions
- Watchdog failsafe writes
- Resume recovery paths

2. Potential timing conflict pattern
- Manual write path completes
- monitor/evaluation/automation path executes shortly after with different intent
- UI observers receive state from mixed timing windows

## Dispatcher Load Map
### Major dispatcher pressure hubs
1. MainViewModel
- High concentration of BeginInvoke/Invoke usage for mode updates, tray action feedback, safe mode events, hotkeys, and service callbacks

2. TrayIconService
- Frequent BeginInvoke update paths for fan/perf/link/health/popup sync

3. OsdService
- Uses Dispatcher.Invoke for show/hide/set mode/update settings/retry hotkey registration

4. HardwareMonitoringService
- Pushes sample collection updates through dispatcher path when not in low-overhead mode

## Architectural Risk Findings
1. Competing state authorities remain
- MainViewModel, GeneralViewModel, TrayIconService, QuickPopupWindow, OsdOverlayWindow each retain mutable mode state caches

2. Polling/timer ownership is fragmented
- Multiple independent loops/timers with partially overlapping domains

3. EC ownership is not fully centralized
- Fan writes are serialized in FanService, but performance EC writes and fan policy writes have separate coordination domains

4. Runtime orchestration is UI-coupled
- MainViewModel coordinates many runtime flows and queue workers that are not purely UI concerns

5. Background/minimized behavior still depends on several visible-only updater surfaces
- Runtime correctness and display freshness can diverge when visibility changes rapidly

## Architecture-Level Remediation Plan (Reduction First)
### Phase A: Authoritative runtime state engine
1. Create RuntimeStateEngine service as sole owner of:
- Fan mode/profile runtime state
- Performance mode runtime state
- Link state
- Last confirmed telemetry snapshot + freshness status

2. Convert all UI surfaces to read-only subscribers:
- TrayIconService
- OsdService/OsdOverlayWindow
- QuickPopupWindow
- DashboardViewModel
- GeneralViewModel
- MainViewModel tab-specific projections

3. Route all mutations through intent API:
- RequestFanModeChange
- RequestPerformanceModeChange
- RequestQuickProfile
- RequestAutomationApply
- RequestHotkeyAction

### Phase B: Polling/timer consolidation
1. Introduce RuntimePollingCoordinator service
- Owns cadence policy and polling order
- Replaces duplicate timer-driven refresh in UI-facing components

2. Move visibility-only timers to pure presentation concerns
- UI timers should never own authoritative refresh decisions

### Phase C: EC operation centralization
1. Introduce EcOperationCoordinator service
- Single serialized queue for EC-bound operations (fan + power limits + future RGB if EC-backed)
- Deterministic operation ordering and cooldown policy
- Shared verification + telemetry cache hooks

2. Keep FanService logic but execute hardware writes through coordinator

### Phase D: MainViewModel reduction
1. Move runtime queue and orchestration responsibilities out of MainViewModel
2. Keep MainViewModel as intent-dispatch + projection adapter
3. Remove duplicate mode-state ownership fields where runtime engine already owns state

## Testing Priorities for Architecture Phase
1. Concurrent state mutation tests
- tray + hotkey simultaneous mode requests
- automation + hotkey overlap
- tray + automation overlap

2. Polling overlap tests
- prove deterministic ordering when monitoring + automation + profile apply overlap

3. Fan overwrite tests
- prove manual fan writes are not immediately overridden without explicit intent

4. Background/minimized lifecycle tests
- prove runtime state engine remains authoritative and fresh with all UI surfaces hidden

5. Dispatcher congestion tests
- simulate high event throughput; assert runtime state correctness and bounded queueing

## Classification Guidance
- Symptom-level stabilization:
  - UI formatting, per-surface cache normalization, isolated lifecycle patches
- Architecture-level remediation:
  - ownership reduction, scheduler consolidation, EC coordinator, runtime-state centralization

This map should be treated as the baseline for architecture reduction PRs.
