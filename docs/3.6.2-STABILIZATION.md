# OmenCore 3.6.2 Stabilization Plan

## Goal
Ship a thermal-safety and runtime-correctness patch set, not a feature release.

## Field Issues Addressed
1. CPU temperature spike when the app window is focused.
2. Broken fan hotkey cycle and phantom fan states.
3. Extreme mode underperforming Gaming during escalation.
4. Auto instability and preset transition RPM collapse.
5. Custom oscillation and custom preset confusion.
6. Battery OSD snapping to 100% during throttled reads.
7. Poor custom-curve UX and mode-label drift.
8. Performance profile confusion between Gaming and Extreme.

## Runtime Corrections
- Single runtime authority is reinforced by preferring service-owned mode state over UI-local mode assumptions.
- Hotkey cycle identity is now deterministic and canonical.
- Extreme mode applies immediately so it can bypass smoothing during a thermal escalation.
- Battery polling now returns the last valid reading during cooldown instead of a placeholder value.
- Focused-window telemetry cadence is reduced to lower dispatch and binding churn.

## Verification Targets
- Fan hotkey cycle should resolve in this exact order: Auto -> Gaming -> Extreme -> Custom -> Quiet.
- Gaming must resolve as Gaming, not Performance or Extreme.
- Quiet must resolve as Quiet, not Custom.
- Extreme curve must remain at or above Gaming on shared temperature points.
- Battery cooldown must surface the last cached charge value.

## Release Criteria
- No phantom fan mode states in runtime mode resolution.
- No battery OSD snap-back to 100 during throttled reads.
- No regression in Extreme ramp-up behavior.
- No increase in focused-window telemetry churn relative to the new cadence baseline.
