# OmenCore v3.6.2 Performance Triage Guide

**Release**: v3.6.2  
**Purpose**: Guide field validation, diagnostics gathering, and performance issue investigation  
**Audience**: Field beta testers, support staff, performance reviewers  

---

## Overview

v3.6.2 introduced architectural improvements to reduce UI/telemetry overhead:
- **Telemetry decoupling**: RuntimeStateEngine controls fan/perf independently from telemetry pipeline
- **Dashboard dormancy**: Telemetry projection suppressed when dashboard minimized (30+ seconds)
- **Hidden-surface suppression**: GeneralView projection suppressed when tab is not visible
- **Render-state caching**: Tray icon and popup render states cached to reduce duplicate renders
- **Cadence reduction**: Active telemetry polling reduced from 1s to 2s for focused-window overhead
- **Latest-wins dispatching**: Hotkey/tray commands coalesced to prevent ping-pong

This guide helps validate these improvements and diagnose performance issues.

---

## Section 1: Baseline Expectations

### Expected Performance Profiles (v3.6.2)

#### Focused Idle (OmenCore window visible, no interaction)
- **CPU**: ≤ 2% (single-threaded polling + projection)
- **GPU**: ≤ 1% (thermal graph rendering)
- **Memory**: ~100-150 MB (stable)
- **Dispatcher calls/min**: 2-4 (timer-driven at ~2s)
- **Active cadence**: 2s (v3.6.2 reduction from 1s)

#### Unfocused Idle (OmenCore minimized but running)
- **CPU**: ≤ 1% (tray-only polling, minimal projection)
- **GPU**: ≤ 0.5% (tray icon only)
- **Memory**: ~100-150 MB (stable)
- **Dispatcher calls/min**: 1-2 (tray update cadence)

#### Tray-Only Idle (OmenCore window not running)
- **CPU**: ≤ 0.5% (tray monitoring only)
- **GPU**: negligible (<0.1%)
- **Memory**: N/A (tray process only)

#### Dashboard Dormant (minimized 30+ seconds)
- **CPU**: -30 to -50% reduction vs focused idle
- **GPU**: -50 to -80% reduction
- **Reason**: Telemetry projection suppressed, dashboard receives latest sample only

#### Gaming Session (perf mode active)
- **CPU (OmenCore)**: ≤ 3% (no UI active, cadence reduced)
- **GPU (OmenCore)**: ≤ 0.5% (no thermal graph rendering)
- **Dispatcher calls/min**: 2-4 (perf preset polling)
- **Fan ramping**: Smooth and responsive (≤ 5s)

#### Long Idle (2+ hours, screen on/off)
- **Thermal**: Stable within ±5°C
- **Fan**: No erratic ramping or chattering
- **CPU trend**: Flat (no slow degradation)
- **Memory**: Stable (no leaks)

## Section 2: Diagnostics Gathering

### Part A: Real-Time Performance Capture

#### Tools Available
- **Windows Task Manager**: CPU, GPU, memory, disk
- **Windows Performance Monitor**: Real-time counters, advanced profiling
- **OmenCore Diagnostics Export**: Built-in counters (dispatcher, projection, thermal)
- **GPU-Z**: Real-time GPU utilization and temperature
- **HWiNFO**: Real-time thermal and fan RPM
- **ETW Trace**: Deep system profiling (advanced)

#### Quick Capture Protocol
1. **Baseline capture** (1 min): OmenCore focused, idle
   - Record: CPU %, GPU %, memory, dispatcher calls, projection calls
   - Goal: Establish normal performance

2. **Dormancy capture** (5 min): OmenCore minimized, idle
   - Record: CPU %, GPU %, dispatcher calls
   - Compare to baseline
   - Goal: Verify 30-50% CPU reduction

3. **Tab switch capture** (1 min): Switch away from dashboard, back
   - Record: CPU spike on switch, recovery time
   - Goal: Verify ≤ 50ms spike

4. **Hotkey capture** (1 min): Rapid fan/perf mode changes
   - Record: Dispatcher queue depth, command coalescing
   - Goal: Verify no ping-pong, no command loss

#### How to Export OmenCore Diagnostics
1. Open OmenCore
2. Navigate to **Diagnostics** section (if available)
3. Export to CSV or JSON
4. Include: dispatcher amplification ratio, projection acceptance ratio, hidden-surface suppression stats, cadence mode, tray/popup suppression status

#### Bounded Performance Snapshot Mode (v3.6.2)
- Diagnostics export now includes `runtime-performance-bounded.txt` with a bounded sample window (3 samples, 5s interval).
- Use `OMENCORE_DIAGNOSTIC_SCENARIO` to label the scenario before exporting diagnostics.
- Example scenario labels: `focused-idle`, `tray-idle`, `dashboard-active`, `popup-active`, `osd-active`, `screen-off-idle`.
- Purpose: capture cheap, high-signal evidence without enabling expensive continuous logging.

Bounded snapshot now includes triage-ready classes:
- `DispatcherAmplificationClass` / `ProjectionAmplificationClass` (`Excellent`, `Expected`, `Elevated`, `Critical`)
- `MainProjectionAcceptanceClass` / `DashboardAcceptanceClass` / `GeneralAcceptanceClass`
- `TrayRenderCacheClass` / `PopupRenderCacheClass`
- `EstimatedCpuClass`
- `ScenarioAssessment` summary line for quick operator triage

Runtime context in bounded output now includes:
- monitoring health + source
- current cadence reason + low-overhead mode
- fan control state (curve/hold activity)

### Part B: Log Analysis

#### Windows Event Logs
- Location: **Event Viewer > Windows Logs > System**
- Filter for OmenCore errors or warnings
- Look for: driver issues, permission denials, hardware verification failures

#### OmenCore Application Logs
- Location (Windows): `C:\Users\<username>\AppData\Local\OmenCore\logs\`
- Location (Linux): `~/.local/share/omencore/logs/`
- Key patterns to search:
  - **Cadence changes**: Look for "active cadence" or "polling mode" entries
  - **Projection gating**: Look for "projection suppressed" or "dormancy" entries
  - **Dispatcher activity**: Look for "queue depth" or "coalesced" entries
  - **Hardware errors**: Look for "fan sanity", "preset verify", "verification deferred" entries
  - **Thermal issues**: Look for spike or ramp patterns

#### Performance Counter Logs
- OmenCore may export performance counters to log files
- Parse for: amplification ratios, acceptance ratios, skip counts, suppression stats

### Part C: Thermal & Fan Validation

#### Thermal Baseline
1. **Idle thermal** (15 min): OmenCore idle, screen on
   - Record: Starting temp, final temp, fan behavior
   - Expected: ±5°C stability
   
2. **Load thermal** (10 min): Gaming or CPU-intensive workload
   - Record: Peak temp, thermal curve, fan response
   - Expected: Smooth, responsive fan ramping

3. **Screen-off idle thermal** (10 min): OmenCore idle, screen off
   - Record: CPU, GPU, temp trend, fan behavior
   - Expected: No thermal amplification vs screen-on idle

#### Fan Ramping Validation
- **Fast ramp**: Fan responds within 5 seconds to thermal load
- **Smooth ramp**: No chattering or erratic speed changes
- **Correct preset**: Fan speed matches selected fan mode (Silent, Auto, Aggressive)
- **No lockup**: Fan maintains steady state, no sudden stops

## Section 3: Interpreting Diagnostics

### Dispatcher Amplification Ratio

#### What It Measures
Number of dispatcher invocations per user-initiated command.
- **Ratio = 1.0**: One dispatch per command (perfect coalescing)
- **Ratio = 1.5**: 1.5 dispatches per command (acceptable)
- **Ratio = 2.0**: Two dispatches per command (expected overhead)
- **Ratio > 3.0**: Possible queue issue or command loss

#### High Amplification (> 3.0) Indicates
- Rapid overlapping commands not properly coalesced
- Possible ping-pong between fan modes or perf presets
- Latest-wins queue may not be functioning correctly
- May correlate with "mode drift" or "hotkey desync" reports

#### How to Diagnose
1. Capture dispatcher logs during rapid hotkey spam (10+ commands in 1 second)
2. Verify queue depth never exceeds 2
3. Verify final applied state matches final command
4. If amplification > 2.5, collect ETW trace for detailed queue analysis

---

### Projection Acceptance Ratio

#### What It Measures
Percentage of telemetry samples that resulted in UI projection (visual update).
- **Ratio = 100%**: Every sample projected (high render pressure)
- **Ratio = 50%**: Every other sample projected (normal throttling)
- **Ratio = 10%**: One in ten projected (strong suppression)

#### High Acceptance (> 80%) Indicates
- Dashboard dormancy not suppressing telemetry
- Possibly hidden-surface suppression failed
- May correlate with "0.3fps UI" or "thermal graph stutter"

#### Low Acceptance (< 10%) Indicates
- Strong suppression (good for idle)
- If during focused dashboard: possible projection gate issue

#### How to Diagnose
1. Capture projection logs during focused dashboard use (should be 80-100%)
2. Capture projection logs during minimized/dormant use (should be 10-30%)
3. If dormant ratio > 50%, verify dashboard dormancy threshold is reached
4. If focused ratio < 50%, check for unintended suppression

---

### Dashboard Dormancy Statistics

#### What To Expect
- **Dormancy threshold**: 30+ seconds minimized before suppression kicks in
- **CPU reduction**: 30-50% when dormant
- **GPU reduction**: 50-80% when dormant
- **Wake latency**: ≤ 100ms from user action to full telemetry projection

#### Dormancy Not Occurring Indicates
- Dashboard visibility gate may be broken
- Dormancy timer may have reset unexpectedly
- Possible unintended surface visibility

#### How to Diagnose
1. Minimize OmenCore
2. Wait 31 seconds
3. Check projection logs for "dormancy activated" or equivalent
4. Monitor CPU before/after dormancy activation
5. If CPU not reduced, verify visibility gate in code

---

### Hidden-Surface Suppression Statistics

#### What To Expect
- **General tab hidden**: 0-5% projection acceptance (suppressed)
- **General tab visible**: 80-100% projection acceptance (normal)
- **CPU reduction when hidden**: 5-15% (hidden-surface suppression alone)

#### Hidden-Surface Not Suppressing Indicates
- GeneralView visibility binding may be broken
- Projection gate not checking surface visibility
- Possible background render churn

#### How to Diagnose
1. Open Dashboard, navigate to General tab
2. Capture projection logs

---

### Thermal-Limit Saturation at Low Temperature (Field Intake)

#### Symptom Signature
- Thermal-limit reason/indicator remains at 100% while CPU package temperature is only ~40-50C.
- CPU package power appears capped (commonly reported around 20-30W) despite low measured temperature.
- Some reports also include custom fan preset instability (preset resets or intermittent fan shutdown behavior).

#### Initial Classification
- Treat as high-priority field anomaly until disproven.
- Do not classify as normal thermal throttling when measured temperature is far below expected throttling range.

#### Required Evidence Capture
1. Export diagnostics with scenario label `thermal-limit-low-temp` (bounded snapshot enabled).
2. Capture synchronized HWiNFO screenshot/log showing:
   - CPU package temperature
   - Thermal limit reason indicator
   - CPU package power
   - Effective clock
3. Record `MonitoringSource` from diagnostics/log output and include the CPU authority suffix (for example `CPU Authority: WMI BIOS`, `CPU Authority: ACPI Thermal Zone`, or `CPU Authority: LHM Fallback`).
4. Search logs for `CPU thermal authority switched:` entries from [src/OmenCoreApp/Hardware/WmiBiosMonitor.cs](../src/OmenCoreApp/Hardware/WmiBiosMonitor.cs) and attach the surrounding lines to show transition reason and timing.
5. Record active fan preset and whether a custom preset was used within the prior 10 minutes.
6. Record whether issue clears after preset reapply, app restart, or system reboot.

#### Fast Triage Actions
1. Reapply current preset, then switch to Auto and back to previous preset.
2. Verify fan-control command path acknowledges successful apply.
3. If lock persists, collect evidence bundle and escalate as potential EC/firmware interaction issue.

#### Operator Runbook
- For a single-pass reproduction and capture workflow, execute [docs/3.6.2-SCENARIO-A-OPERATOR-RUNBOOK.md](3.6.2-SCENARIO-A-OPERATOR-RUNBOOK.md).

---

### Cadence Mode Statistics

#### Expected States
- **Active cadence**: 2s (v3.6.2 when overlay visible)
- **Tray cadence**: 2-5s (when minimized)
- **Idle cadence**: 5-10s or disabled (when unfocused long-term)
- **Screen-off cadence**: Reduced or disabled (Windows idle)

#### Cadence Not Changing Indicates
- Cadence mode detection may be broken
- Focused/unfocused state tracking may be incorrect
- Possible timer not responding to visibility changes

#### How to Diagnose
1. Capture cadence logs during focused use (verify 2s)
2. Minimize and wait 30s (verify cadence increases or disables)
3. Maximize (verify cadence returns to 2s)
4. Turn off screen and wait (verify cadence reduces)
5. If cadence doesn't change, check overlay visibility event handler

## Section 4: Common Failure Patterns

### Pattern 1: "0.3fps UI" / Heavy Stutter During Idle

**Symptoms**:
- Dashboard becomes very unresponsive when minimized then restored
- Frame rate drops to 0.3fps during thermal graph scroll
- Smooth at first then degrades

**Root Causes**:
- Dashboard dormancy not suppressing projection (projection acceptance > 50% while minimized)
- Thermal graph render churn during dormancy recovery
- GPU memory pressure from accumulated telemetry buffers

**How to Investigate**:
1. Capture projection logs during minimized + restore cycle
2. Verify dormancy suppression is active (acceptance ratio drops)
3. Monitor GPU memory after restore (should not spike)
4. Check for render queue buildup during dormancy
5. Look for "projection gating disabled" or similar errors in logs

**Evidence to Collect**:
- Before/after projection acceptance ratio
- GPU memory usage before/after dormancy
- Frame time histogram during graph scroll
- Render queue depth logs

---

### Pattern 2: Focused CPU Spike / Focused Window Churn

**Symptoms**:
- CPU spikes to 5-10% when OmenCore is focused
- Spikes correlate with overlay visibility changes
- GPU usage higher than expected

**Root Causes**:
- Cadence not properly reducing from 1s to 2s
- Projection acceptance too high (not throttling)
- Hidden-surface suppression not working for General tab

**How to Investigate**:
1. Capture cadence logs (should show 2s while active)
2. Capture projection logs (should show throttling at 50-60%)
3. Monitor dispatcher call frequency (should be ~30/min at 2s cadence)
4. Check hidden-surface suppression during General tab navigation
5. Look for "cadence override" or "projection forced" messages

**Evidence to Collect**:
- Cadence mode logs
- Dispatcher call frequency graph
- Projection acceptance ratio during focused use
- CPU usage trend during 1-minute focused idle
- GPU utilization logs

---

### Pattern 3: "Thermal Amplification" During Idle

**Symptoms**:
- Idle temperature climbs faster than expected (>1°C per minute)
- Temperature doesn't stabilize; continues climbing during 30-min idle
- Fan ramping aggressively despite idle workload
- Temperature drops rapidly when user moves mouse

**Root Causes**:
- Telemetry cadence not reducing during idle (stuck at 2s instead of 5-10s)
- Dispatcher continuously waking from sleep
- Thermal graph render occurring even while minimized
- Timer not responding to focus/idle state transitions

**How to Investigate**:
1. Capture thermal logs during 30-min idle session
2. Verify temperature trend (should plateau within 5 min)
3. Capture cadence logs (should show reduced cadence)
4. Monitor CPU usage trend (should remain ≤ 1%)
5. Check for "cadence disabled" or similar idle indicators
6. Look for dispatcher wake calls in logs
7. Verify tray-only mode when minimized

**Evidence to Collect**:
- 30-min thermal trend (with timestamps)
- CPU usage trend (with timestamps)
- Cadence state changes (with timestamps)
- Dispatcher call frequency during idle
- Fan speed graph during idle
- Any "wake from idle" or "cadence reset" log messages

---

### Pattern 4: "Hot Key Desync" / Mode Not Applied

**Symptoms**:
- Hotkey pressed but mode doesn't change (or changes after delay)
- Mode changes then reverts unexpectedly
- Final mode doesn't match final hotkey press
- Rapid hotkey spam causes mode oscillation

**Root Causes**:
- Latest-wins queue not functioning (commands not coalesced)
- Dispatcher queue full or dropping commands
- State event not firing after command applied
- Command not reaching runtime controller

**How to Investigate**:
1. Enable dispatcher logging
2. Perform rapid hotkey spam (10+ commands in 2 seconds)
3. Capture dispatcher queue depth logs (should not exceed 2)
4. Capture final applied state (should match final command)
5. Verify no command loss in logs
6. Check for "queue overflow" or "command dropped" errors
7. Look for ping-pong pattern in state change events

**Evidence to Collect**:
- Dispatcher queue depth graph during hotkey spam
- Command sequence log showing all enqueued commands
- Applied state sequence log
- State change event times
- Any "queue full" or "dropped" error messages

---

### Pattern 5: Popup/Tray Render Churn / CPU Spike When Tray Active

**Symptoms**:
- CPU spikes to 3-5% when tray tooltip visible or popup open
- Tray updates very frequently (CPU usage pattern shows regular spikes)
- Popup displays briefly then disappears (render loop issue)
- GPU usage higher than expected for tooltip display

**Root Causes**:
- Render-state cache not functioning (every sample re-rendered)
- Projection acceptance too high for tray (should be low, only on change)
- Popup render loop triggered constantly (not throttled)
- Tooltip text updated on every sample (not cached)

**How to Investigate**:
1. Enable tray render logging
2. Monitor popup display time (should be throttled to 1-2s)
3. Capture render state cache hits/misses (high miss rate = problem)
4. Monitor GPU usage while popup visible
5. Look for "render cache miss" or "cache invalidated" entries
6. Verify popup is truly only re-rendered on data change

**Evidence to Collect**:
- Tray render call frequency graph
- Popup display duration logs
- Render-state cache hit/miss ratio
- GPU usage during popup display
- Render frequency vs. data change frequency comparison

---

### Pattern 6: Fan Instability / Erratic Ramping

**Symptoms**:
- Fan speeds oscillate or chatter (ramping up/down rapidly)
- Fan doesn't respond smoothly to temperature changes
- Fan sometimes doesn't ramp when thermal load increases
- Fan mode changes don't take effect immediately

**Root Causes**:
- Mode not applied correctly via dispatcher
- Thermal readings oscillating (sensor noise)

---

### Pattern 7: RGB Dynamic Effects Unsupported On Specific Platforms

**Symptoms**:
- Static color applies successfully, but breathing/spectrum/rainbow effects do not apply.
- Logs show provider-level unsupported-effect messages without app crash.

**Root Causes**:
- Platform/provider exposes only static effect support.
- Dynamic effect request fanout includes providers that do not advertise the requested effect.

**How to Investigate**:
1. Capture `RGB sync` logs during dynamic effect apply.
2. Confirm unsupported providers are explicitly skipped.
3. Verify static color still applies to the same provider set.
4. Classify as graceful degradation unless static effects also fail.

**Evidence to Collect**:
- Provider list from diagnostics status output.
- Logs showing `unsupported` skip lines and final sync result.
- A/B result: static effect success vs dynamic effect failure on same run.

## Section 5: Final RC Failure Classification

Use this fast classification after each field bundle:

1. **Blocker**
   - thermal-limit saturation at 40-50C reproduced
   - persistent 20-30W package lock not cleared by preset reapply + app restart
   - fan stop/reset under custom preset stress
2. **High**
   - dispatcher/projection amplification repeatedly in `Critical`
   - hidden-surface suppression absent in minimized/tray scenarios
3. **Medium**
   - cache-hit classes degraded (`Watch` or `Low`) with no user-visible regression
   - isolated elevated CPU class without thermal drift
4. **Low**
   - cosmetic wording/layout issues only

Escalation rule:
- Any **Blocker** result holds 3.6.2 release and requires either targeted fix or explicit rollback decision.
- Preset verification failing (hardware issue)
- Cadence causing delayed thermal response

**How to Investigate**:
1. Capture fan speed logs at high frequency (1Hz+)
2. Capture thermal reading logs (check for oscillation)
3. Monitor mode change events (verify applied immediately)
4. Check for "preset verify failed" entries
5. Look for "fan mode override" or "hardware error" messages
6. Verify cadence is not suppressing thermal feedback

**Evidence to Collect**:
- Fan speed graph (1-second granularity) during thermal load
- Thermal reading graph (check for noise)
- Mode change event log
- Preset verification attempt logs
- Hardware error or warning entries

## Section 5: Collecting & Attaching Evidence

### Minimum Evidence for Performance Bug Report

When reporting a performance issue, include:
1. **System Profile**
   - OS version and build number
   - OmenCore version
   - Hardware model (if applicable)
   - GPU driver version
   - BIOS/firmware version

2. **Reproduction Steps**
   - Exact actions to reproduce the issue
   - Time to reproduction (immediate, 30 min, 2+ hours)
   - Reproducibility (always, often, rarely)

3. **Baseline Metrics**
   - CPU usage before issue occurs
   - GPU usage before issue occurs
   - Temperature before issue occurs
   - Fan speed before issue occurs

4. **Issue Metrics**
   - CPU usage during issue
   - GPU usage during issue
   - Temperature during issue
   - Duration of issue

5. **Diagnostics Logs**
   - OmenCore application logs (last 10 min)
   - Dispatcher logs if available
   - Projection logs if available
   - Cadence/visibility state logs

6. **System Logs**
   - Windows Event Viewer errors/warnings
   - GPU driver logs if available
   - Thermal management logs

7. **Screenshots/Video**
   - Screenshot of Task Manager showing CPU/GPU spike
   - Video of fan speed changing erratically (if applicable)
   - Screenshot of OmenCore UI state when issue occurs

---

### Optional Evidence (For Deep Analysis)

- **ETW Trace** (advanced):
  - CPU sampling trace during issue reproduction
  - GPU trace (if supported by driver)
  - File I/O trace to check for disk contention

- **Performance Monitor Capture**:
  - Real-time CPU, GPU, disk, network counters
  - Custom counters (if OmenCore exposes them)

- **Thermal/Fan Logs**:
  - Full thermal curve during issue reproduction
  - Fan speed graph at 1-second granularity

## Section 7: Troubleshooting Guide

### Issue: "Can't find OemenCore logs"
- **Windows**: Check `C:\Users\<username>\AppData\Local\OmenCore\logs\`
- **Linux**: Check `~/.local/share/omencore/logs/`
- **If missing**: Enable diagnostics export in settings or check application output

### Issue: "Dispatcher logs show no activity"
- Dispatcher logging may not be enabled by default
- Check OmenCore settings for "Diagnostics" or "Debug" options
- May require rebuild with DEBUG configuration

### Issue: "Projection acceptance always 100%"
- Projection throttling not enabled
- Check dashboard visibility gate in code (DashboardViewModel.cs)
- Verify dormancy timer is running

### Issue: "Cadence always shows as active"
- Cadence state tracking may be broken
- Check polling coordinator for visibility change events
- Verify window focus events firing correctly

### Issue: "Can't measure GPU usage"
- Windows Task Manager may not show per-app GPU
- Use GPU-Z or vendor-specific monitoring tool
- May require GPU driver update

## Section 8: Next Steps

After collecting evidence:

1. **Categorize**: Determine if issue is OmenCore-specific or environmental
2. **Reproduce**: Attempt to reproduce in lab environment with same hardware
3. **Isolate**: Disable specific features (dormancy, suppression, etc.) to isolate root cause
4. **Fix**: Implement targeted fix based on root cause
5. **Validate**: Re-test with same reproduction steps
6. **Release**: Deploy fix to beta testers or stable release
