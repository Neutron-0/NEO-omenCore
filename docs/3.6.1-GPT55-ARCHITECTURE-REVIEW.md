# OmenCore v3.6.1 GPT-5.5 Architecture Review

## Executive Summary
OmenCore v3.6.1 is materially safer than the v3.6.0 regression baseline, but the runtime is still in a hybrid architecture state. The new RuntimeStateEngine, RuntimePollingCoordinator, RuntimeCommandDispatcher, RuntimeHotkeyCoordinator, and RuntimeEcOperationCoordinator are useful reduction points, yet several old UI-owned and service-owned paths still mutate runtime state or cadence directly.

The immediate release-readiness risk is not one obvious missing feature. It is partial migration: new coordinators exist, but some legacy paths still bypass them. That creates two kinds of hazards: false confidence from centralization that is not actually complete, and race windows where old surfaces still behave as state owners.

## Current Architecture Health
- Fan and performance apply paths are substantially improved through service event synchronization, latest-wins tray/hotkey dispatch, and shared fan/performance name resolvers.
- RuntimeStateEngine currently acts as a projection distributor, not as the sole runtime authority. MainViewModel still owns the mutable fan/performance projection that feeds it.
- RuntimePollingCoordinator owns several cadence transitions. A follow-up slice now routes DashboardViewModel and SettingsViewModel low-overhead writes through the coordinator during normal app runtime, with direct service fallback retained only for isolated construction/tests.
- RuntimeEcOperationCoordinator serializes major FanService, PerformanceModeService, FanCleaningService, V1 keyboard EC, V2 keyboard EC-direct, PowerVerificationService, SystemControl EC GPU boost fallback, and diagnostic export EC-read paths.

## Hybrid Architecture Risks
- Projection authority remains split between FanService, PerformanceModeService, MainViewModel fields, DashboardViewModel fields, GeneralViewModel fields, TrayIconService caches, QuickPopupWindow caches, and OSD overlay fields.
- MainViewModel is still a service locator and runtime constructor for most of the app. This makes DI-level singleton sharing incomplete, especially for coordinators that should be global.
- Some UI surfaces are read-only subscribers in spirit but still contain local caches or direct apply methods; the GeneralViewModel telemetry timer and obsolete MainViewModel fan-curve bypass have now been removed.

## Remaining Major Risks
- A mode change can still be represented by multiple local caches before the hardware-confirmed service event fully converges.
- EC-direct keyboard writes, SystemControl EC GPU boost fallback, and diagnostic export EC reads were routed through the shared coordinator in recent review slices. Remaining EC risk is now focused on less-common hardware/control paths outside the audited fan/performance/keyboard/power-verification/system-control/diagnostic-export set.
- Settings/Dashboard low-overhead toggles were routed through RuntimePollingCoordinator in this review slice. Remaining polling risk is broader timer fragmentation, not this specific bypass.
- Full-suite test duration remains an architecture signal: the runtime has many timers, loops, and background workers even when the current evidence no longer shows a hard deadlock.

## Runtime Ownership Findings
- FanService remains the correct fan authority.
- PerformanceModeService remains the correct performance authority.
- HardwareMonitoringService remains the telemetry authority.
- MainViewModel still duplicates fan/performance state and constructs runtime services directly, which blocks clean singleton ownership.
- GeneralViewModel still applies paired fan/performance profiles directly, but no longer owns a telemetry timer; its telemetry projection is now fed from the normalized monitoring sample stream.
- DashboardViewModel owns presentation mode fields and a 1s uptime timer; its telemetry stream is event-driven, but low-overhead mutation is not fully coordinator-routed.

## Polling/Timer Findings
- The polling coordinator is only a first slice. It centralizes cadence-affecting calls from MainViewModel/App lifecycle, and DashboardViewModel/SettingsViewModel low-overhead writes now use it during normal runtime. Broader timer fragmentation still remains.
- QuickPopupWindow uses a visible-only 1s presentation timer. This is acceptable as presentation work, but it should remain a subscriber only.
- GeneralViewModel's 1s DispatcherTimer has been removed. It now updates fan RPM/percent, temperatures, load, power, and memory projection from `UpdateFromMonitoringSample`, leaving it closer to a subscriber surface instead of a polling owner.
- WmiFanController countdown extension was a material CPU risk during custom curve hold: skipped ticks still called `ExtendFanCountdown()`, which reissued fan-level WMI writes. The current slice makes skipped manual/preset ticks no-ops, bounds reapply cadence, and throttles remaining countdown-extension writes, but longer-term timer ownership should still move out of backend-specific ad hoc timers.
- WmiBiosMonitor fallback CPU-temperature reads now use capped timeout backoff after repeated stalls. This reduces fixed-cadence retry pressure in HWiNFO/contention scenarios, but the fallback provider is still an ad hoc monitor side path and should eventually be folded into a single sensor-source arbitration policy.

## Dispatcher Findings
- MainViewModel remains the largest dispatcher pressure hub. It queues monitoring updates, fan/performance service events, hotkey actions, safe-mode events, and lifecycle transitions.
- RuntimeStateEngine subscriber dedupe in App.xaml.cs reduces repeated tray/OSD mode updates, but the engine is still fed by MainViewModel property changes instead of service events directly.

## EC Coordination Findings
- FanService, PerformanceModeService, FanCleaningService, and KeyboardLightingService V1 EC writes are partially coordinated.
- KeyboardLightingServiceV2 EcDirectBackend now routes color, brightness, backlight, effect, and readback operations through RuntimeEcOperationCoordinator.
- PowerVerificationService now routes power-limit apply and readback through RuntimeEcOperationCoordinator.
- SystemControlViewModel now routes EC GPU boost detection/apply/startup-reapply register operations through RuntimeEcOperationCoordinator.
- DiagnosticExportService now routes support-bundle EC register reads through RuntimeEcOperationCoordinator, reducing diagnostic capture contention with live runtime operations.

## Concurrency Findings
- RuntimeCommandDispatcher latest-wins behavior addresses the v2.9.0 tray freeze regression class.
- Profile sync guards reduce linked-mode ping-pong, but linked fan/performance sync still occurs from MainViewModel event handlers and should move into a non-UI runtime service later.
- Coordinator instances are currently created inside MainViewModel instead of DI. This works for services constructed there, but it is fragile and easy to bypass.

## Release Blockers
- No hard release blocker identified for the release-candidate packaging pass.
- The highest-confidence pre-release bypass closures are complete and documented: Dashboard/Settings low-overhead polling, V2 keyboard EC direct, power verification, SystemControl EC GPU boost fallback, GeneralViewModel telemetry timer removal, MainViewModel fan-curve bypass removal, and DiagnosticExportService EC reads.
- Remaining architecture-completeness work is non-blocking and moved to post-3.6.1 consolidation.

## Systems That Should Be Removed
- MainViewModel-owned runtime construction should be removed in a post-slice refactor and replaced with DI-owned singleton runtime services.
- GeneralViewModel's 1s telemetry polling timer has been removed; the remaining post-release consolidation is to feed it from RuntimeStateEngine directly instead of via MainViewModel's normalized sample handoff.
- Duplicate fan/performance mode caches in UI surfaces should be reduced to presentation-only state fed by RuntimeStateEngine snapshots.

## Systems That Should Be Consolidated
- Low-overhead, overlay realtime, window-active, and tray-only cadence writes should all flow through RuntimePollingCoordinator.
- All EC-bound write paths, including keyboard V2 EC direct and diagnostic power verification writes, should flow through RuntimeEcOperationCoordinator.
- Fan/performance link orchestration should move out of MainViewModel into a runtime coordination service after v3.6.1 unless a safe pre-release slice is available.

## Recommended Pre-Release Tasks
- Completed in this review slice: route SettingsViewModel and DashboardViewModel low-overhead mode changes through RuntimePollingCoordinator.
- Completed in this review slice: inject RuntimeEcOperationCoordinator into KeyboardLightingServiceV2 and EcDirectBackend, and wrap EC-direct keyboard writes/readback sequences.
- Completed in follow-up high-CPU slice: reduce WMI fan keepalive write cadence and bound automatic curve post-apply verification.
- Completed in follow-up high-CPU slice: remove steady manual fan-write RPM snapshots and add adaptive CPU-temperature fallback timeout backoff.
- Completed in final hardening slice: remove GeneralViewModel's independent telemetry timer and obsolete MainViewModel fan-curve bypass.
- Completed in final hardening slice: route SystemControl EC GPU boost fallback and DiagnosticExportService EC register reads through RuntimeEcOperationCoordinator.
- Completed in release-candidate prep: version metadata, installer fallback version, README/install references, and RC checklist are aligned to v3.6.1.
- Remaining pre-release activity is validation and packaging only; no further architecture refactor is recommended unless validation finds a hard blocker.

## Recommended Post-3.6.1 Tasks
- Move MainViewModel service construction into DI.
- Make RuntimeStateEngine subscribe directly to FanService, PerformanceModeService, and HardwareMonitoringService.
- Move GeneralViewModel telemetry projection from MainViewModel's normalized sample handoff to a direct RuntimeStateEngine subscription.
- Replace per-surface fan/performance caches with snapshot-bound presentation state.
- Add long-running minimized/tray-only integration tests that assert cadence, state freshness, and clean teardown.

## Overall Release Readiness Assessment
v3.6.1 is materially release-ready as a focused stabilization candidate, but should not be described as fully architecturally consolidated. It is currently a stabilized hybrid with the highest-confidence pre-release bypasses closed, release-gate coverage added for the latest reductions, Release build validation passing with 0 warnings/0 errors, and the full Windows test suite passing 623/623. Remaining hybrid risks are documented and should be treated as post-release consolidation work unless new field evidence identifies a concrete blocker.
